## From Imperative to Functional

学计算机的各位可能都知道图灵机(Turing Machine)，~~也可能不知道~~，因为现在的电脑都是冯诺伊曼架构的(这是现实中图灵机的实现)。但其实还有一种计算模型，由Alonzo Church提出，叫做λ演算(Lambda Calculus)，但由于其不好实现且跟图灵机是等价的，就不怎么为人所知。图灵机讲计算类比(抽象)为一条无限长的纸条和读写头等，λ演算讲计算类比为函数，这也是命令式和函数式的区别。
- 命令式(Imperative)告诉计算机怎么做，最小单元是语句/命令
- 函数式(Functional)，或者叫声明式(Declarative)，告诉计算机做什么，最小单元是函数

啊，这俩有什么区别吗，命令式里不是也有函数的概念吗？其实命令式的函数是过程(procedure)，并且其借鉴了函数式的抽象思想。学过计算机组成原理的同学都知道，函数调用的本质是直接跳转和返回(附带一些压栈弹栈的操作)，所以命令式的函数并不是真正意义上的函数。函数式的函数在理念上有点极端
- 函数没有副作用(Side Effect)，即函数的输出只依赖于输入，不依赖于外部状态
- 这就导致了函数式没有变量的概念，因为变量会引入状态，也因此没有堆的概念
- ~~把所有变量都换成常量，就相当于在写函数式~~

这就不得不提到函数式的一些优点了，也说明了为什么要学习函数式
- 天然的并行性和线程安全，因为没有状态/全局变量，所以不会有竞态条件
- 更容易测试，因为函数没有副作用，所以可以更容易debug
- 更容易证明正确性和优化，因为这些严谨规则和定义可以用数学分析
- 先天支持模式匹配，适合处理复杂的数据结构，也适合做编译器(这个可能没那么直观)

当然函数式也有一些缺点，比如
- 现实中需要IO操作，而函数式的理念与其相悖，所以就需要一个包装(Wrapper)来~~欺骗自己~~处理IO
- 难写难读，因为没有那么直观，比如没有循环；另一方面这也算半个优点，因为很难写出屎山代码

其实命令式借鉴了许多函数式的思想，比如函数的抽象、递归等。不过说到底，这俩都是形式和系统，都是从什么公理什么假设什么原语出发推导出来的，没有好坏之分，只有适合和不适合。它们也不是相互取代的关系，而都是为了更好地写出优雅的代码。目前就没有"纯"函数式和命令式的语言，都是混合的。

## Function, Function and Function
